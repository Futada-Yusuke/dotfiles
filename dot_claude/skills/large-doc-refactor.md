# 大規模ドキュメント修正スキル

## 概要
複数の大規模な技術ドキュメント（設計書、要件定義書等）を効率的に修正するための手順とベストプラクティス

## 適用シーン
- システム設計の方針転換（例: Data API → 通常のDB接続）
- アーキテクチャの大幅な変更
- 複数ドキュメント間の整合性確保が必要な場合
- 1000行以上の大規模ドキュメントの修正

## 基本原則

### 1. サブエージェントの活用
大規模な修正はサブエージェントに委譲してコンテキストを節約

```
良い例: Task toolでgeneral-purposeエージェントを起動
悪い例: メインコンテキストで全ての修正を実行
```

### 2. 段階的アプローチ
一度にすべてを修正せず、段階的に実施

```
Phase 1: 影響範囲の特定と優先順位付け
Phase 2: 主要な変更（アーキテクチャ図、技術スタック等）
Phase 3: 詳細な変更（コード例、設定値等）
Phase 4: 整合性チェック
```

### 3. バックアップの作成
大規模な置換を行う前に必ずバックアップ

```bash
cp original.md original.md.backup
```

## 修正手順

### Step 1: 影響範囲の分析

**サブエージェントに依頼する内容**:
```
以下のドキュメント群で、[変更内容]に関連する箇所をリストアップしてください:
- doc1.md
- doc2.md
- doc3.md

変更内容: [具体的な変更内容を記載]
```

### Step 2: 変更計画の作成

**記録すべき内容**:
- 削除する概念・技術（例: Data API, SQS, jobsテーブル）
- 追加する概念・技術（例: Drizzle ORM, NAT Gateway）
- 影響を受けるセクション
- 修正の優先順位

**Note**: 要件整理・設計段階では、変更履歴ではなく「技術選定の背景」を記載する

### Step 3: 技術選定の背景記載

各ドキュメントに以下の形式で背景を追記:

```markdown
#### [技術名]採用の背景

**選定理由**:
- 理由1
- 理由2
- 理由3

**[代替技術]を採用しなかった理由**:
- 理由1
- 理由2

**コスト影響**:
- 項目1: +/- $XX/月
- 合計: +/- $XX/月
```

**Note**: 意思決定が増えてドキュメントが肥大化する場合は、**Step 3.5: 事実と意思決定の分離**を検討する。

### Step 3.5: 事実と意思決定の分離（オプション）

ドキュメントに意思決定の背景が多数含まれ、可読性が低下している場合、ADR（Architecture Decision Records）への分離を検討する。

#### 分離の判断基準
- 意思決定の背景が3つ以上ある
- 各背景の説明が10行以上
- ドキュメントの総行数が1000行以上
- 意思決定の履歴を残す必要がある

#### 分離の手順

1. **decisions/ディレクトリ作成**
   ```bash
   mkdir -p decisions
   ```

2. **ADRファイル作成**
   - 命名規則: `NNN_short_description.md`
   - 例: `001_mysql_3306_over_data_api.md`

3. **意思決定の背景を抽出してADR化**
   - 技術仕様書から「選定理由」「背景」セクションを抽出
   - ADR形式に変換（詳細は`adr-management.md`スキル参照）

4. **技術仕様書の簡潔化**
   - 意思決定の背景セクションを削除
   - ADRへの参照リンクに置き換え

   削除前:
   ```markdown
   #### VPC内配置とNAT Gateway採用の背景

   **選定理由**:
   - ORM使用のためVPC内配置が必須
   - 外部API接続のためNAT Gateway必要

   **コスト影響**: $64/月
   ```

   削除後:
   ```markdown
   ### MySQL 3306接続方式

   VPC内配置、NAT Gateway 2個使用。

   **意思決定の背景**: [ADR-001](./decisions/001_mysql_3306_over_data_api.md)、[ADR-003](./decisions/003_vpc_nat_gateway.md)
   ```

5. **decisions/README.md作成**
   - ADR一覧表を作成
   - 各ADRへのリンクと概要を記載

**メリット**:
- 技術仕様書が簡潔になり、事実（What/How）に集中
- 意思決定の履歴が明確に記録される
- ドキュメントの保守性が向上

**参照**: 詳細は`adr-management.md`スキル参照

### Step 4: サブエージェントによる修正実行

**効果的な依頼方法**:
```
## Task
以下のドキュメントを修正してください:
- doc1.md
- doc2.md

## 主要な修正内容
1. [具体的な修正1]
2. [具体的な修正2]
3. [具体的な修正3]

## 注意事項
- 既存の内容は削除せず、適切な箇所に追記してください
- 各ドキュメントの文脈に合わせて表現を調整してください
- 冗長にならないよう、簡潔に記載してください

## 期待する成果
修正後のファイルを保存し、主要な変更点をサマリーとして報告してください。
```

### Step 5: 整合性チェック

**サブエージェントに依頼する内容**:
```
以下のドキュメント間の整合性を確認し、矛盾があれば修正してください:
- doc1.md
- doc2.md

## 確認すべき項目
1. テーブル設計の整合性
2. 技術スタックの整合性
3. 処理フローの整合性
4. 実装例の整合性

## 矛盾が見つかった場合
矛盾箇所を修正し、最新の意思決定内容で統一してください。
```

## ベストプラクティス

### DO（推奨）
✅ 大規模修正はサブエージェントに委譲
✅ 変更前にバックアップを作成
✅ 技術選定の背景を明記
✅ 段階的に修正を実施
✅ 最後に整合性チェックを実施
✅ 簡潔で明確な記述

### DON'T（非推奨）
❌ メインコンテキストで全ての修正を実行
❌ 一度にすべてのドキュメントを修正
❌ 変更履歴を詳細に記録（設計段階では不要）
❌ 冗長な説明
❌ 整合性チェックの省略

## トラブルシューティング

### 問題: Editツールで文字列が見つからない

**原因**: 全角/半角の違い、空白文字の違い

**解決策**:
1. バックアップからsedで一括置換
2. サブエージェントに修正を依頼
3. ファイル全体の書き換えを検討

### 問題: ドキュメント間で矛盾が発生

**解決策**:
整合性チェック専用のサブエージェントを起動し、自動修正

### 問題: コンテキストが不足

**解決策**:
- サブエージェントを活用
- 修正を複数フェーズに分割
- 優先度の高いものから順次実施

## サンプルプロンプト集

### パターン1: 技術スタック変更
```
以下のドキュメント群で、Data APIからMySQL 3306接続への変更を反映してください:
- 03_system_design.md
- 05_backend_guide.md
- 04_infrastructure_guide.md

変更内容:
- Data API削除
- Drizzle ORM / SQLAlchemy追加
- VPC内配置、NAT Gateway追加
- 技術選定の背景を各ドキュメントに追記
```

### パターン2: テーブル設計変更
```
jobsテーブルをsales_text_generationsテーブルに統一してください:
- ER図の更新
- データフロー図の更新
- Repository実装例の更新
- 統一の背景を追記
```

### パターン3: 整合性チェック
```
以下の観点でドキュメント間の整合性を確認してください:
1. テーブル名の一致
2. 技術スタックの一致
3. 処理フローの一致
4. 実装例の一致

矛盾があれば修正し、レポートしてください。
```

## まとめ

大規模ドキュメント修正では:
1. **サブエージェントを活用**してコンテキストを節約
2. **段階的に修正**して複雑性を管理
3. **技術選定の背景を明記**して意思決定を記録
4. **整合性チェック**で品質を確保

このアプローチにより、効率的かつ高品質なドキュメント修正が可能になります。
