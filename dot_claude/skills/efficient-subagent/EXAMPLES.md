# サブエージェント活用の具体例

このドキュメントでは、実際のユースケースに基づいたサブエージェントの効果的な使い方を示します。

## 例1: データモデル調査

### ユーザーリクエスト
「OrderとCustomerの関係性を調べてください」

### ❌ 非効率なアプローチ（避けるべき）

直接ツールで順次調査:
1. Grep で "Order" を検索
2. 結果を読む
3. Read でスキーマファイルを読む
4. Grep で "Customer" を検索
5. 結果を読む
6. Read で別のスキーマファイルを読む

**問題点**: コンテキストを大量に消費し、非効率

### ✅ 効率的なアプローチ（推奨）

Exploreサブエージェントを使用して一度に調査:

```
Task tool with subagent_type=Explore
- 目的: OrderとCustomerの関係性を調査
- スコープ: データベーススキーマ、TypeScript型定義、API定義
- Thoroughness: very thorough
```

**メリット**:
- コンテキスト消費量を大幅削減
- 体系的な調査
- 関連情報を一度に取得

## 例2: 複数ドキュメントの一括更新

### ユーザーリクエスト
「要件書と設計書の用語を統一してください（ユーザー → 会員）」

### ❌ 非効率なアプローチ（避けるべき）

1. Read で要件書を読む
2. Edit で1箇所ずつ修正
3. Read で設計書を読む
4. Edit で1箇所ずつ修正

**問題点**: 時間がかかり、修正漏れのリスク

### ✅ 効率的なアプローチ（推奨）

2つのimpl-executorサブエージェントを**並列実行**:

**並列実行の例**:
- サブエージェント1: 要件書の一括更新
- サブエージェント2: 設計書の一括更新

1つのメッセージで両方のTaskを起動することで、効率的に処理

**メリット**:
- 時間短縮
- 一貫した変更
- 変更内容のサマリー自動生成

## 例3: 影響範囲の分析

### ユーザーリクエスト
「ユーザー権限モデルを変更する影響を調査してください」

### ✅ 効率的なアプローチ

複数のExploreサブエージェントを並列実行:

1. **現在の権限システム調査**
   - データベーススキーマ
   - API権限チェックロジック
   - フロントエンドコンポーネント

2. **関連ドキュメント調査**
   - 要件書
   - 設計書
   - API仕様書

**並列実行により、調査時間を大幅に短縮**

---

## 例4: 新機能の実装

### ユーザーリクエスト
「新しい権限テーブルを追加し、API、ドキュメントを更新してください」

### ✅ 効率的なアプローチ

複数のサブエージェントを段階的に使用:

**フェーズ1: 計画立案**
- task-plannerで実装タスクを分解

**フェーズ2: 実装（並列実行）**
- impl-executor 1: データベーススキーマ追加
- impl-executor 2: API権限チェック実装
- impl-executor 3: ドキュメント更新

**フェーズ3: レビュー**
- code-reviewerで実装をレビュー

**メリット**:
- 包括的な調査

- 構造化されたレポート生成

## 例6: ドキュメント整合性の確認と修正（実践例）

### ユーザーリクエスト
「アグリゲート設計のドキュメント更新漏れを調査し、修正してください」

### ✅ 効率的なアプローチ（実践済み）

**フェーズ1: 並列調査（3つのExploreサブエージェント）**

1つのメッセージで3つのタスクを同時実行:
- Task 1: `docs/` 配下のアグリゲート設計更新漏れを調査 (Explore, medium)
- Task 2: `specs/` 配下のアグリゲート設計更新漏れを調査 (Explore, medium)
- Task 3: `guides/` 配下のアグリゲート設計更新漏れを調査 (Explore, medium)

**フェーズ2: 優先度順の修正**

調査結果に基づいて、優先度の高い順に修正:
1. backend_guide.md のRepository一覧を修正
2. backend_guide.md のCascade削除実装パターンを修正
3. database_design.md にアグリゲート設計セクションを追加
4. api_design.md の削除方式記載を明確化
5. architecture.md のCascade削除コード例を更新

**成果**:
- 並列調査により、調査時間を大幅に短縮
- 複数ドキュメント間の不整合を体系的に発見
- 優先度順の修正により、重要な箇所から対応
- 複数のドキュメントを効率的に更新完了

**学び**:
- 独立した調査タスクは必ず並列実行
- 調査結果を基に修正の優先順位を判断
- ドキュメント間の整合性を常に検証

## サブエージェントタイプ早見表

| タスク | サブエージェント | 理由 |
|--------|---------------|------|
| コードベース探索 | Explore | 体系的な調査、コンテキスト効率化 |
| ドキュメント更新 | impl-executor | 一括変更、フォーマット保持 |
| 技術調査 | researcher | 外部情報収集、包括的調査 |
| 実装計画 | task-planner | タスク分解、優先順位付け |
| コードレビュー | code-reviewer | 品質チェック、問題発見 |
| 複数ファイル実装 | impl-executor（並列） | 時間短縮、一貫性保持 |
