# Efficient Subagent Usage

## 目的

調査や実装の変更タスクにおいて、サブエージェント（Task tool）を効率的に活用し、コンテキストの使用量を最適化する。

## いつこのスキルを使うか

以下のようなタスクで**必ず**このスキルを適用する:

1. **コードベースの調査**
   - 機能の実装場所を探す
   - データモデルや関係性を理解する
   - 既存コードの動作を分析する

2. **ドキュメントの更新**
   - 要件定義書の修正
   - 設計書の更新
   - 複数ファイルの一括変更

3. **実装タスク**
   - 新機能の追加
   - バグ修正
   - リファクタリング

4. **影響範囲の分析**
   - 変更による影響箇所の特定
   - 依存関係の調査

## 基本原則

### 1. コンテキスト効率化の優先

直接ツール（Read, Grep, Glob）を使う前に、以下を検討する:

- **探索的タスク**: Task tool with `subagent_type=Explore` を使用
- **実装タスク**: Task tool with `subagent_type=impl-executor` を使用
- **調査タスク**: Task tool with `subagent_type=researcher` を使用

### 2. 適切なサブエージェントの選択

| タスクの種類 | 使用するサブエージェント | 理由 |
|------------|---------------------|------|
| コードベース探索 | `Explore` | コンテキストを節約しながら体系的に調査 |
| ドキュメント更新 | `impl-executor` | 複数の変更を効率的に実行 |
| 技術調査 | `researcher` | 外部情報も含めた包括的な調査 |
| 計画立案 | `task-planner` | タスクの分解と優先順位付け |
| コードレビュー | `code-reviewer` | 実装後の品質チェック |

### 3. 並列実行の活用（最重要）

**独立したタスクは必ず並列実行する** - これがコンテキスト効率化の最大のポイント:

**悪い例**: 順次実行（避けるべき）
1. ファイルAを調査（Task tool使用）
2. 結果を待つ
3. ファイルBを調査（Task tool使用）
4. 結果を待つ

→ **問題点**: 待機時間が発生し、非効率。コンテキストも無駄に消費。

**良い例**: 並列実行（推奨）
1つのメッセージで複数のTask tool呼び出しを同時実行:
```
単一のfunction_callsブロック内で複数のTask toolを呼び出し:
- Task 1: ファイルAの調査 (Explore)
- Task 2: ファイルBの調査 (Explore)
- Task 3: ドキュメントCの更新 (impl-executor)
```

→ **メリット**: 並列実行により時間を大幅に短縮、コンテキスト効率化

**並列実行の判断基準**:
- タスク間に依存関係がない → 並列実行
- 一方のタスクの結果が他方に必要 → 順次実行

## 実装ガイドライン

### ステップ1: タスクの分析

タスクを受け取ったら、以下を判断する:

1. このタスクはサブエージェントで効率化できるか？
2. どのサブエージェントが最適か？
3. 並列実行できる部分はあるか？

### ステップ2: プロンプトの設計

サブエージェントへのプロンプトは具体的に:

- **目的**: 何を達成するか明確に記述
- **スコープ**: 調査・変更の範囲を指定
- **期待する出力**: どのような形式で結果を返すか

### ステップ3: 実行と結果の活用

サブエージェントの結果を受け取ったら:
- 結果を要約してユーザーに報告
- 次のアクションを提案
- 必要に応じて追加のサブエージェントを起動

## 例: 調査タスク

```
ユーザー: データモデルを調査してください

1. Task tool (subagent_type=Explore) を使用
   - データベーススキーマを探索
   - モデル定義ファイルを検索
   - 関連するテーブルとリレーションを特定
   - Thoroughness level を指定

2. 結果をユーザーに報告
```

## 例: ドキュメント更新タスク

```
ユーザー: 要件書を最新化してください

1. まず変更内容を分析（必要に応じてExploreで調査）
2. Task tool (subagent_type=impl-executor) を使用
   - 変更箇所をリストアップ
   - 一括で編集を実行
   - マークダウンフォーマットを保持

3. 変更サマリーをユーザーに報告
```

## 例: 複数ファイルの並列処理

```
ユーザー: 要件書と設計書の両方を更新してください

1つのメッセージで2つのTask toolを同時起動:
- Task 1: 要件書の更新（impl-executor）
- Task 2: 設計書の更新（impl-executor）

並列実行により時間を短縮
```

## 避けるべきパターン

### ❌ 直接Readで大量のファイルを読む
```
ユーザー: この機能がどこに実装されているか調べて
→ Read tool で複数ファイルを順次読む（コンテキスト浪費）
```

### ✅ Exploreサブエージェントを使う
```
ユーザー: この機能がどこに実装されているか調べて
→ Task tool (Explore) で効率的に探索
```

### ❌ 小さな変更を順次実行
```
ユーザー: 10個のファイルの用語を修正して
→ Edit tool で1ファイルずつ修正（非効率）
```

### ✅ impl-executorで一括実行
```
ユーザー: 10個のファイルの用語を修正して
→ Task tool (impl-executor) で一括処理
```

## 並列実行のベストプラクティス

### パターン1: 複数ディレクトリの調査

**ユースケース**: プロジェクト全体の整合性チェック

```
1つのメッセージで3-5個のExploreサブエージェントを起動:
- docs/ 配下の調査
- .kiro/steering/ 配下の調査
- .kiro/specs/ 配下の調査
- packages/backend/ の調査
- packages/frontend/ の調査
```

**効果**: 順次実行に比べて調査時間を80%削減

### パターン2: ドキュメント一括更新

**ユースケース**: 用語統一、構造変更の反映

```
1つのメッセージで複数のimpl-executorを起動:
- ドキュメント1の更新
- ドキュメント2の更新
- ドキュメント3の更新
```

**注意点**: 相互参照がある場合は順次実行を検討

### パターン3: 調査→分析→実装の段階的並列化

**ユースケース**: 複雑な機能実装

```
フェーズ1: 並列調査
- 既存実装の調査 (Explore)
- 関連ドキュメントの調査 (Explore)
- 技術スタックの調査 (researcher)

フェーズ2: 並列実装（調査結果を基に）
- コード実装 (impl-executor)
- ドキュメント更新 (impl-executor)
- テスト作成 (impl-executor)

フェーズ3: レビュー
- コードレビュー (code-reviewer)
```

**効果**: 各フェーズ内で並列実行することで、全体の時間を最小化

### 並列実行の判断フローチャート

```
タスクを受け取る
    ↓
複数の独立したサブタスクに分解できるか？
    ├─ YES → 依存関係をチェック
    │         ├─ 依存関係なし → 並列実行（推奨）
    │         └─ 依存関係あり → 順次実行 or フェーズ分け
    └─ NO  → 単一のサブエージェント実行
```

## チェックリスト

タスクを実行する前に確認:

- [ ] このタスクはサブエージェントで効率化できるか？
- [ ] 適切なサブエージェントタイプを選択したか？
- [ ] **並列実行できるタスクを見落としていないか？（最重要）**
- [ ] プロンプトは具体的で明確か？
- [ ] ユーザーへの報告内容を準備したか？
- [ ] 依存関係のないタスクを順次実行していないか？
- [ ] 各サブエージェントに適切なthoroughness levelを指定したか？
