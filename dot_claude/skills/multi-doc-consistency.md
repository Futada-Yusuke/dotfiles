# 複数ドキュメント間の整合性維持プロセス

## 目的
複数の関連ドキュメントを更新する際に、用語・定義・設計の一貫性を確実に維持する。

## 適用シーン

### このスキルを使うべき状況
- 複数のドキュメントが相互参照している
- 要件定義が設計書・実装ガイドに伝播する構造
- ドキュメント間で用語・テーブル定義・API仕様が統一されている必要がある
- 1つの変更が5つ以上のドキュメントに影響する

### 例
- 要件定義書 → システム設計書 → 実装ガイド（複数）
- API仕様書 → フロントエンド実装 → バックエンド実装
- データモデル定義 → ER図 → Repository実装 → 画面設計

## 4ステップのプロセス

### Step 1: 変更内容の整理と確認

**質問による明確化**:
```
以下の観点で変更内容を確認してください：

1. **スコープ**: どのエンティティ/機能に影響するか？
2. **用語変更**: 名称変更があるか？（例: requests → generations）
3. **データ構造変更**: テーブル追加・削除・カラム変更があるか？
4. **処理フロー変更**: シーケンスやタイミングの変更があるか？
5. **技術スタック変更**: 使用ライブラリ・フレームワークの変更があるか？
```

**確認リスト**:
- [ ] 変更の起点となるドキュメントを特定
- [ ] 変更理由を明確化
- [ ] 影響範囲の仮説を立てる
- [ ] 不明点をすべて質問で解消

### Step 2: 影響範囲の分析

**Exploreエージェントの活用**:
```
Task tool (subagent_type: Explore) を使用して各ドキュメントを分析：

指示内容:
- [対象ドキュメント]で[変更キーワード]が記載されている箇所を特定
- 各箇所で具体的に何を修正すべきか概要を抽出
- 優先度（高/中/低）を判定

例:
"system_design.md で ORDERS テーブルの記載箇所を特定し、
削除が必要な箇所と、ORDER_ITEMS.statusへの統合が必要な箇所を列挙してください"
```

**分析観点の例**:
| 観点 | 検索キーワード | 確認内容 |
|------|--------------|---------|
| ロール定義 | admin, user, manager | 全ドキュメントでロール名が統一されているか |
| テーブル名 | ORDERS, CUSTOMERS, PRODUCTS | ER図、Repository、APIで同じテーブル名か |
| データ階層 | organization > team > project | 全ドキュメントで階層構造が一致しているか |
| API仕様 | /api/v1/orders | フロントエンドとバックエンドでパスが一致しているか |

**出力例**:
```
【system_design.md】
- 優先度: 高
- 修正箇所:
  1. ER図: INVOICES エンティティを削除
  2. テーブル詳細定義: INVOICES セクションを削除
  3. データフロー図: 請求処理の流れを ORDERS.status に統合

【backend_guide.md】
- 優先度: 高
- 修正箇所:
  1. Repository層: InvoiceRepository を削除
  2. API: GET /api/v1/invoices を削除
  3. ディレクトリ構成: invoiceRepository.ts を削除

【frontend_guide.md】
- 優先度: 中
- 修正箇所:
  1. データフェッチング: useInvoices フックを削除
  2. 画面: InvoiceList コンポーネントを削除
```

### Step 3: 並列更新（サブエージェント活用）

**並列実行の原則**:
```
1つのメッセージで複数のTaskツールを呼び出し、
複数ドキュメントを並列で更新する

理由:
- 効率的（待ち時間削減）
- 各エージェントが独立して作業可能
- TodoListで進捗を一元管理
```

**実装パターン**:
```
// 4つのドキュメントを並列更新する例

Task (subagent_type: impl-executor) × 4

Agent 1: system_design.md を修正
- ER図から INVOICES を削除
- テーブル詳細定義セクションを削除
- データフロー図を ORDERS.status 中心に書き換え

Agent 2: infrastructure_guide.md を修正
- API Gateway設定から /invoices パスを削除
- Lambda関数リストから invoiceHandler を削除

Agent 3: backend_guide.md を修正
- Repository層から InvoiceRepository を削除
- API実装から該当エンドポイントを削除
- ディレクトリ構成を更新

Agent 4: frontend_guide.md を修正
- データフェッチング層から useInvoices を削除
- 画面一覧から InvoiceList を削除
- ルーティングから該当パスを削除
```

**各エージェントへの指示で明記すべき内容**:
1. **具体的な修正内容**: 「XXXを削除」「YYYをZZZに変更」
2. **変更理由**: 「INVOICES テーブルを削除し、ORDERS.status で管理するため」
3. **既存の文脈・書式の維持**: 「他のセクションの書式を参考に、同じスタイルで記述」
4. **言語**: 「日本語で作成されているため、日本語で記述」

### Step 4: 整合性確認

**自動チェック（並列Grepの活用）**:
```
Grep tool × 複数 を並列実行して用語の統一を確認

例:
1. Grep "INVOICES" (結果: 0件であることを確認)
2. Grep "organization.*>.*team.*>.*project" (階層構造の表記統一を確認)
3. Grep "/api/v1/orders" (APIパスの統一を確認)
```

**手動チェックリスト**:
- [ ] **用語の統一**: 全ドキュメントでロール名・テーブル名が一致
- [ ] **データ構造の一致**: 階層構造が全ドキュメントで同じ
- [ ] **API仕様の一致**: フロントエンドとバックエンドでパス・パラメータが一致
- [ ] **データフロー図の一貫性**: 処理の流れが矛盾なく記載
- [ ] **Repository層の一致**: TypeScript/Python双方で同じ設計パターン
- [ ] **外部キー制約の一致**: ER図とテーブル定義でCASCADE動作が一致

**サブエージェント出力の検証**:
```
サブエージェントが生成した内容が要件に反していないか確認

チェック方法:
1. Read toolで更新後のファイルを確認
2. 重要なセクション（テーブル定義、データフロー図）を抽出
3. 要件と照らし合わせて矛盾がないか確認

実例:
- 要件: "PostgreSQL + Prisma ORM"
- サブエージェント出力: "MySQL + Drizzle ORM" ❌
- → Edit toolで手動修正
```

## よくある整合性エラーとその対策

### エラー1: 用語の不統一
**症状**: あるドキュメントでは `requests`、別のドキュメントでは `generations`

**対策**:
1. Grep で全ドキュメント検索: `Grep "requests" -n`
2. 対象箇所を特定
3. 並列Editで一括置換: `Edit (replace_all: true)`

### エラー2: 外部キー参照の不一致
**症状**: ER図では `order_id` だが、Repository実装では `project_id` を参照

**対策**:
1. Exploreエージェントでテーブル定義を確認
2. 各ドキュメントで外部キー定義を検索
3. 不一致箇所をリストアップ
4. 並列更新で修正

### エラー3: データフロー図と実装の矛盾
**症状**: データフロー図では「EventBridge → Fargate」だが、実装ガイドでは「SQS → Lambda」

**対策**:
1. データフロー図を確定（ユーザーに確認）
2. 実装ガイドを確定図に合わせて更新
3. インフラガイドも同様に更新
4. 全体を再チェック

### エラー4: サブエージェントが要件を無視
**症状**: 明示的に「PostgreSQL使用」と指示したのに、サブエージェントが「MySQL」に戻す

**対策**:
1. **サブエージェント指示を詳細化**: 「PostgreSQL。MySQLは使用しない」
2. **変更理由を明記**: 「JSON型のサポートが必要なため」
3. **更新後に手動確認**: Read toolで重要箇所を確認
4. **必要に応じて手動修正**: Edit toolで正確に修正

### エラー5: DB接続方式の矛盾
**症状**: 設計書では「直接接続 + ORM」、実装ガイドでは「REST API経由」

**対策**:
1. 最新の技術選定を確認
2. 全ドキュメントで以下を統一:
   - 接続方式（直接接続 or API経由）
   - ORM使用有無
   - インフラ構成
3. 技術選定の背景を各ドキュメントに追記
4. 環境変数を統一

### エラー6: テーブル設計の矛盾（削除済みテーブル参照）
**症状**: 設計書では`jobs`テーブル削除、実装ガイドでは`JobRepository`が残存

**対策**:
1. Grepで削除済みテーブル名を全検索
2. 影響箇所をリストアップ:
   - Repository層のクラス削除
   - API エンドポイント削除
   - データフロー図の更新
   - ER図の更新
3. 代替テーブルで統一
4. 並列更新で全箇所を修正

## ドキュメント構造別の対策

### パターン1: 階層型（要件 → 設計 → 実装）
```
例: requirements.md → system_design.md → backend_guide.md → frontend_guide.md

戦略:
1. 要件書を正とする
2. Explore × 3 で設計、バックエンド、フロントエンドの影響箇所を分析
3. impl-executor × 3 で並列更新
4. 要件書の定義と一致しているか確認
```

### パターン2: 相互参照型（設計 ↔ 実装）
```
例: API仕様書 ↔ バックエンド実装 ↔ フロントエンド実装

戦略:
1. API仕様書を正とする
2. バックエンド・フロントエンドの両方を仕様に合わせる
3. Grepで "/api/v1/..." パスを全検索
4. 不一致箇所を並列修正
```

### パターン3: 分散型（複数の独立ドキュメント）
```
例: 各マイクロサービスのREADME、各モジュールのドキュメント

戦略:
1. 共通する用語・定義を抽出
2. Grepで全ドキュメント検索
3. 用語集を作成（単一の正）
4. 各ドキュメントを用語集に合わせる
```

## チェックリスト

### 更新前
- [ ] 変更内容を明確化（推測なし）
- [ ] 影響範囲の仮説を立てた
- [ ] Exploreエージェントで分析完了
- [ ] 並列更新の計画を立てた

### 更新中
- [ ] TodoListで進捗管理している
- [ ] 各サブエージェントへの指示が具体的
- [ ] 変更理由を各エージェントに伝えている
- [ ] 既存の書式・文脈を維持するよう指示

### 更新後
- [ ] 用語が全ドキュメントで統一されている
- [ ] テーブル定義が全ドキュメントで一致
- [ ] データフロー図が矛盾なく記載
- [ ] API仕様がフロント・バックで一致
- [ ] サブエージェント出力を手動確認した

## 注意事項

1. **推測しない**: 不明点は必ず質問で明確化
2. **並列実行を活用**: 独立したドキュメントは並列更新
3. **サブエージェント出力を検証**: 要件に反していないか確認
4. **既存の書式維持**: 各ドキュメントの文脈・スタイルを尊重
5. **段階的更新**: 一度にすべて更新せず、重要度順に実施
6. **技術選定の背景記録**: 設計段階では「なぜその技術を選んだか」を記録（変更履歴は不要）
7. **コスト影響の明記**: インフラ変更時はコスト影響を必ず記載

## 成功の指標

- ✅ すべてのドキュメントで用語が統一されている
- ✅ テーブル定義が全ドキュメントで一致している
- ✅ データフロー図が矛盾なく記載されている
- ✅ 実装ガイドが要件定義に準拠している
- ✅ サブエージェントの誤りをすべて修正した
- ✅ 変更履歴が追跡可能である
