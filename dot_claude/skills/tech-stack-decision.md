# 技術スタック選定の意思決定フレームワーク

## 目的
技術選定時に、サポート状況・コスト・保守性・パフォーマンス・既存資産を総合的に評価し、長期的に持続可能な判断を行う。

## 基本原則

### 推測ではなく根拠ベースの判断
- ❌ 「このライブラリは人気がある」（主観的）
- ❌ 「一般的にこの技術が使われている」（根拠不足）
- ✅ 「GitHubのコミット頻度は過去6ヶ月で月2回以下」（事実）
- ✅ 「公式ドキュメントで"deprecated"と明記されている」（事実）
- ✅ 「コスト試算で月$250の削減効果」（定量的）

### 短期と長期のバランス
- **短期**: 初期開発速度、学習コスト
- **長期**: 保守性、サポート継続性、乗り換えコスト

### トレードオフの明確化
すべての技術選定にはトレードオフがある。メリット・デメリット双方を明確にする。

## 評価の5つの観点

### 1. サポート状況（Support）
**最重要**: サポートが切れた技術は負債になる

#### 確認項目
- [ ] 公式ドキュメントが最新か？
- [ ] GitHubのコミット頻度は？（月何回？）
- [ ] Issue/PRの対応速度は？
- [ ] メンテナが複数いるか？（バス係数）
- [ ] 依存ライブラリのサポート状況は？
- [ ] 過去に"deprecated"になった履歴は？

#### 判定基準
| レベル | 状態 | 例 |
|-------|------|---|
| 🟢 高 | 月10回以上のコミット、活発なコミュニティ | React, TypeScript, Next.js |
| 🟡 中 | 月1-5回のコミット、定期的なバージョンアップ | Hono, Zod |
| 🔴 低 | 月0-1回、最終更新が6ヶ月以上前 | Kysely Data API Client |

#### 実例: Kyselyの判断
```
Kysely Data API Client:
- GitHubコミット: 過去6ヶ月で3回のみ
- 最終リリース: 1年前
- Issue未解決: 15件
- メンテナ: 1名のみ

判定: 🔴 サポート切れリスクが高い
→ 採用見送り、生SQL + Data API SDKに変更
```

### 2. コスト（Cost）
**重要**: 初期コストだけでなく、運用コストも考慮

#### 確認項目
- [ ] 初期構築コスト（人件費含む）
- [ ] 月次運用コスト（AWS料金等）
- [ ] スケール時のコスト増加率
- [ ] 代替技術との比較

#### 判定基準
| 項目 | 判定基準 |
|------|---------|
| 初期 | 学習コスト × 開発期間 |
| 運用 | 月次料金 × 12ヶ月 × 3年 |
| スケール | トラフィック2倍時のコスト増加率 |

#### 実例: ECS Fargate vs Lambda（バッチ処理）
```
想定: 月間100万トランザクション、実行時間15分/タスク

Lambda:
- 初期: $0（学習コスト低）
- 運用: $313.14/月
- スケール: トラフィック2倍で$626/月

ECS Fargate (Python):
- 初期: $500（既存Workerコード活用）
- 運用: $50-60/月
- スケール: タスク数2倍で$100-120/月

3年間総コスト:
- Lambda: $11,273
- ECS Fargate: $2,160 + $500初期 = $2,660

判定: ECS Fargate採用で3年で約$8,600削減
```

#### 実例: NAT Gateway削除の効果
```
Aurora Serverless V2 + Data API:
- VPC不要 → NAT Gateway削除
- コネクションプール不要 → RDS Proxy削除

削減額:
- NAT Gateway: $64/月
- RDS Proxy: $13.2/月
- 合計: $77.2/月 = $2,779/年

判定: Data API採用で年間$2,779削減
```

### 3. 保守性（Maintainability）
**重要**: コードの理解しやすさ、変更のしやすさ

#### 確認項目
- [ ] ドキュメントが充実しているか？
- [ ] コードが読みやすいか？（抽象化レベル）
- [ ] デバッグが容易か？
- [ ] チーム全員が理解できるか？
- [ ] 将来の技術乗り換えが容易か？

#### 判定基準
| 観点 | 良い例 | 悪い例 |
|------|-------|-------|
| 抽象化 | 生SQL（可読性高） | 複雑なORMクエリ（魔法的） |
| 依存関係 | 標準SDK（AWS SDK） | サードパーティラッパー |
| 可搬性 | 生SQLは移植容易 | ORM依存は移植困難 |

#### 実例: 生SQL + Data API SDK vs ORM
```
生SQL + Data API SDK:
✅ SQLが直接読める（可読性高）
✅ AWS SDKのみ依存（依存最小）
✅ Kysely/Drizzleが衰退しても影響なし
✅ 将来活発なライブラリに乗り換え容易

ORM (Kysely/Drizzle):
❌ クエリビルダーの学習コスト
❌ サードパーティライブラリ依存
❌ ライブラリ衰退時に大規模書き換え
❌ Data APIサポート状況に依存

判定: 生SQL + Data API SDK採用で長期保守性向上
```

### 4. パフォーマンス（Performance）
**中程度**: 要件を満たす範囲で最適化

#### 確認項目
- [ ] レイテンシ要件を満たすか？
- [ ] スループット要件を満たすか？
- [ ] スケーラビリティはあるか？
- [ ] ボトルネックはどこか？

#### 判定基準
| 要件レベル | 目標 | 適切な技術 |
|----------|------|----------|
| 低レイテンシ（<100ms） | API応答時間 | Lambda + API Gateway |
| 高スループット（>1000 req/s） | 並列処理 | ECS Fargate（マルチプロセス） |
| 長時間実行（>15分） | バッチ処理 | ECS Fargate, Step Functions |

#### 実例: Data APIの制約
```
Data API:
- 制約: 64KBまでの行サイズ
- 制約: 同期実行のみ（非同期不可）

影響評価:
✅ トランザクションデータは64KB未満（問題なし）
✅ バッチ処理は同期で十分（問題なし）
❌ 大容量BLOB保存は不可（→ S3に分離）

判定: 要件を満たすため、Data API採用可能
```

### 5. 既存資産活用（Asset Reuse）
**中程度**: 既存コード・知見の活用でコスト削減

#### 確認項目
- [ ] 既存コードを流用できるか？
- [ ] チームの既存スキルを活かせるか？
- [ ] 既存インフラと統合できるか？
- [ ] 新規学習コストは許容範囲か？

#### 実例: Python Workerの活用
```
既存資産:
- Python 3.xで実装されたスクレイピングWorker
- Anthropic Claude APIクライアント
- マルチプロセス並列化ロジック

選択肢:
1. TypeScriptで全書き換え
   - 初期: 2週間（書き換え）+ 1週間（テスト）
   - リスク: バグ混入の可能性

2. Pythonのまま活用（ECS Fargate）
   - 初期: 3日（ECS化）
   - リスク: 低（既存コード活用）

判定: Python + ECS Fargate採用で開発期間10日短縮
```

## 意思決定プロセス

### Step 1: 要件の明確化
```
以下を明確にする：
1. 機能要件: 何を実現するか？
2. 非機能要件: パフォーマンス、可用性、セキュリティ
3. 制約条件: コスト上限、開発期間、チームスキル
4. 優先順位: どの要件が最重要か？
```

### Step 2: 候補技術のリストアップ
```
観点別に候補を列挙：
- 既存で使っている技術
- 業界標準の技術
- 新興の有望な技術
- 代替となる技術
```

### Step 3: 5観点での評価
```
各候補を5観点で評価：
| 技術 | サポート | コスト | 保守性 | パフォーマンス | 既存資産 | 総合 |
|------|---------|-------|-------|--------------|---------|------|
| A    | 🟢      | 🟡    | 🟢    | 🟢           | 🔴      | 🟢   |
| B    | 🔴      | 🟢    | 🔴    | 🟡           | 🟢      | 🔴   |
```

### Step 4: トレードオフの明確化
```
選択肢Aのメリット:
- サポートが活発
- 保守性が高い
- パフォーマンス要件を満たす

選択肢Aのデメリット:
- 初期コストが高い
- 既存資産を活かせない

選択肢Bのメリット:
- 既存資産を活かせる
- コストが安い

選択肢Bのデメリット:
- サポートが不安定
- 保守性が低い
```

### Step 5: 判断と記録
```
判断:
選択肢Aを採用

理由:
1. サポート継続性が最重要（長期運用を考慮）
2. 初期コストは許容範囲（予算内）
3. 既存資産は段階的に移行可能

代替案が必要になる条件:
- サポートが6ヶ月以上停止した場合
- 運用コストが予算の2倍を超えた場合
```

## 実践例

### 例1: ORM選定
```
要件:
- Aurora Serverless V2 + Data API使用
- TypeScript/Python双方対応
- 長期運用（3年以上）

候補:
1. Kysely + Data API Client
2. Drizzle ORM
3. Prisma (マイグレーションのみ) + 生SQL + Data API SDK

評価:
| 技術 | サポート | コスト | 保守性 | パフォーマンス | 既存資産 | 総合 |
|------|---------|-------|-------|--------------|---------|------|
| Kysely | 🔴 (月1回未満コミット) | 🟢 | 🟡 | 🟢 | 🔴 | 🔴 |
| Drizzle | 🟡 (Data API対応不明) | 🟢 | 🟡 | 🟢 | 🔴 | 🟡 |
| Prisma + 生SQL | 🟢 (Prismaサポート活発) | 🟢 | 🟢 (可搬性高) | 🟢 | 🔴 | 🟢 |

判断: Prisma (マイグレーションのみ) + 生SQL + Data API SDK採用

理由:
1. サポート: Prismaは活発、AWS SDKも安定
2. 保守性: 生SQLは可搬性が高く、ライブラリ乗り換え容易
3. リスク回避: Kysely Data API Clientのサポート切れリスクを回避
```

### 例2: バッチ処理の実行環境
```
要件:
- スクレイピング + AI生成の長時間実行（15分）
- 月間100万トランザクション処理
- 既存PythonWorkerコードを活用

候補:
1. Lambda (TypeScript書き換え)
2. ECS Fargate (Python既存コード)

評価:
| 技術 | サポート | コスト | 保守性 | パフォーマンス | 既存資産 | 総合 |
|------|---------|-------|-------|--------------|---------|------|
| Lambda | 🟢 | 🔴 ($313/月) | 🟡 (書き換え必要) | 🟡 (15分制限) | 🔴 (書き換え) | 🔴 |
| ECS Fargate | 🟢 | 🟢 ($50-60/月) | 🟢 (既存コード) | 🟢 (時間制限なし) | 🟢 (活用可) | 🟢 |

判断: ECS Fargate (Python) 採用

理由:
1. コスト: Lambda比80-90%削減（$250/月削減）
2. 既存資産: Pythonコードをそのまま活用、開発期間10日短縮
3. パフォーマンス: 15分制限なし、マルチプロセス並列化可能
4. 保守性: 既存チームのPythonスキルを活かせる
```

### 例3: フロントエンドフレームワーク
```
要件:
- SPA (Single Page Application)
- TypeScript
- 高速なビルド
- チームが習熟している技術

候補:
1. Next.js (App Router)
2. Vite + React + TanStack Router

評価:
| 技術 | サポート | コスト | 保守性 | パフォーマンス | 既存資産 | 総合 |
|------|---------|-------|-------|--------------|---------|------|
| Next.js | 🟢 | 🟢 | 🟢 | 🟡 (ビルド遅い) | 🟢 (既存スキル) | 🟢 |
| Vite + React | 🟢 | 🟢 | 🟢 | 🟢 (高速ビルド) | 🟡 (学習必要) | 🟢 |

判断: Vite + React + TanStack Router採用

理由:
1. パフォーマンス: Viteのビルド速度がNext.js比10倍高速
2. 柔軟性: SPAに特化、不要な機能（SSR）を含まない
3. 保守性: シンプルな構成、依存関係が少ない
4. サポート: Vite, React, TanStackすべて活発
```

## よくある失敗パターンと対策

### 失敗1: 「人気がある」だけで選定
**症状**: GitHubスター数だけで判断

**対策**:
- コミット頻度を確認
- Issue/PRの対応速度を確認
- メンテナの人数を確認
- 依存ライブラリのサポート状況を確認

### 失敗2: 初期コストのみで判断
**症状**: 学習コストを無視して新技術を採用

**対策**:
- 3年間の総コスト（初期 + 運用）を試算
- チームの習熟度を考慮
- 既存資産の活用可能性を評価

### 失敗3: サポート切れリスクの無視
**症状**: マイナーなライブラリを深く依存

**対策**:
- 月次コミット頻度を確認
- メンテナのバス係数を確認（1名のみは危険）
- 代替技術への乗り換えコストを事前評価

### 失敗4: 過剰な抽象化
**症状**: ORMやフレームワークで魔法的に実装

**対策**:
- 生のSQL/APIを理解できるレベルの抽象化にとどめる
- デバッグが容易な実装を優先
- 「魔法」ではなく「明示的」な実装

## チェックリスト

### 選定前
- [ ] 要件を明確化した（機能・非機能・制約）
- [ ] 候補技術を3つ以上リストアップした
- [ ] 各候補を5観点で評価した
- [ ] トレードオフを明確にした

### 選定後
- [ ] 判断理由を文書化した
- [ ] 代替案が必要になる条件を定義した
- [ ] チームに共有した
- [ ] 定期的に再評価する計画を立てた（6ヶ月毎等）

## 注意事項

1. **推測しない**: 「たぶん大丈夫」ではなく、事実ベースで判断
2. **短期と長期を両立**: 初期コストだけでなく、3年後を考慮
3. **サポート継続性を最重視**: サポート切れは技術的負債の最大要因
4. **トレードオフを明確化**: すべてが完璧な技術は存在しない
5. **判断を記録**: なぜその技術を選んだか、後で振り返れるように

## 成功の指標

- ✅ 3年後もサポートが継続している
- ✅ 運用コストが予算内に収まっている
- ✅ チーム全員が技術を理解している
- ✅ 技術的負債が増えていない
- ✅ 乗り換えが必要になった際、容易に移行できる
