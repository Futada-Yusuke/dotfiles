# codex mcpを使ったコードレビュー

## 目的
codex mcpを活用して、変更内容を体系的にレビューし、潜在的な問題を早期に発見する

## 適用タイミング
- プルリクエスト作成前
- 大きな機能実装が完了した時
- リファクタリング後
- ユーザーが明示的にレビューを依頼した時

## 手順

### 1. 変更内容の確認

まず、変更の規模と影響範囲を把握する：

```bash
# 変更ファイルの一覧と統計
git diff [base-branch]...[feature-branch] --stat

# 変更行数の確認
git diff [base-branch]...[feature-branch] | wc -l
```

### 2. codex mcpでレビュー実行

mcp__codex__codex ツールを使用してレビューを実行：

```
プロンプト例：
このブランチ([branch-name])での変更内容をレビューしてください。

# レビュー対象
[base-branch]ブランチから[feature-branch]ブランチへの変更内容

# 変更ファイル
[変更されたファイルのリスト]

# レビュー観点
- コードの品質（可読性、保守性）
- セキュリティ上の問題
- パフォーマンスへの影響
- エラーハンドリングの適切性
- テストカバレッジ
- 後方互換性
- ベストプラクティスへの準拠

変更内容を確認して、問題点や改善提案があれば指摘してください。
```

**重要**: `approval-policy: never` を設定してレビューのみ実行

### 3. レビュー結果の分析

codex mcpの出力から以下を確認：

- **Blocking issues**（重大な問題）: 必ず修正が必要
- **設計上の問題**: 意図を確認してから対応を判断
- **パフォーマンスへの影響**: 実測して判断
- **提案・改善案**: 優先度を判断して対応

### 4. 指摘事項への対応

#### 重大な問題（Blocking issues）
1. 問題箇所を特定
2. 修正方針を決定
3. 修正を実装
4. テストで検証

#### 設計上の問題
1. 元々の設計意図を確認
   - 過去の実装履歴を調査（`git log`, `git show`）
   - 既存の使用箇所を確認（`grep`, `Grep`ツール）
2. 修正が必要か判断
3. 必要なら修正を実装

#### パフォーマンス関連
1. 実測可能な場合は計測
2. ベンチマークテストを追加
3. 修正前後で比較

### 5. 修正後の検証

```bash
# テスト実行
pytest [test-files] -v

# 全テスト実行
pytest [test-directory] -v

# 修正内容をコミット
git add [modified-files]
git commit -m "fix: [レビュー指摘への対応内容]"
```

### 6. 再レビュー（必要に応じて）

大きな修正を行った場合は、再度codex mcpでレビューを実行

## ベストプラクティス

### レビュー観点のカスタマイズ

プロジェクトやチームの方針に応じて観点を追加：

- **言語固有の観点**:
  - Python: PEP 8準拠、型ヒント
  - TypeScript: 型安全性
- **フレームワーク固有**:
  - React: Hooks使用方法
  - Lambda: コールドスタート対策
- **プロジェクト固有**:
  - 独自のコーディング規約
  - アーキテクチャパターン

### 効果的なレビュー

- **小さい単位でレビュー**: 大きなPRは複数回に分けてレビュー
- **コンテキストを提供**: 変更の背景や目的を説明
- **優先順位をつける**: 全ての指摘に対応する必要はない

### レビュー結果の活用

- **パターンの発見**: 同じ問題が繰り返される場合はスキル化・ルール化
- **チーム共有**: 有益な指摘はチームで共有
- **継続的改善**: レビュー観点を定期的に見直し

## 注意事項

- codex mcpはコードを実行しないため、**ロジックの正しさは別途テストで検証**
- レビュー結果は**参考として活用**し、最終判断は人間が行う
- **既存の設計意図**を確認せずに盲目的に修正しない
- セキュリティに関する指摘は**特に慎重に対応**
