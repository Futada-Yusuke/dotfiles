# 要件明確化のための質問テンプレート

## 目的
曖昧な要件や不完全な情報に対して、推測を避け、構造化された質問で明確化する。

## 基本原則

### 推測禁止の徹底
- ❌ 「おそらく〜だと思います」
- ❌ 「一般的には〜です」
- ❌ 「類似の機能では〜」
- ✅ 「〜について確認させてください」
- ✅ 「以下の理解で正しいでしょうか？」

### 質問の段階的詳細化
1. **全体像の確認**: データ構造、処理フロー、役割定義
2. **詳細の確認**: テーブル設計、API仕様、ビジネスロジック
3. **エッジケースの確認**: エラーハンドリング、リトライ、例外処理

## 質問テンプレート集

### 1. データ構造の確認

#### データ階層の図示
```
以下の理解で正しいでしょうか？

[親エンティティ]
  ├─ [子エンティティA]
  │   └─ [孫エンティティ]
  └─ [子エンティティB]

この階層で、各エンティティ間のリレーションは以下でよろしいですか？
- 親 : 子A = 1 : N
- 子A : 孫 = 1 : N
- 親 : 子B = 1 : N
```

#### テーブル設計の確認
```
[テーブル名]について確認させてください：

1. 主キー: [予想されるカラム名] でよろしいですか？
2. 外部キー: [親テーブル]との関連は CASCADE DELETE でよろしいですか？
3. ステータス管理: status カラムの取りうる値は何でしょうか？
4. タイムスタンプ: created_at, updated_at 以外に必要な日時カラムはありますか？
   (例: sent_at, completed_at, deleted_at等)
5. NULL許可: [特定カラム]はNULL許可でよろしいですか？理由は？
```

### 2. 処理フローの確認

#### 段階的な処理確認
```
処理フローを段階的に確認させてください：

Phase 1: [処理名]
- トリガー: [イベント名]
- 入力: [データソース]
- 出力: [結果の格納先]
- エラー時: [どうする？]

Phase 2: [処理名]
- トリガー: [Phase 1の結果？それとも別のイベント？]
- 入力: [Phase 1の出力を使う？]
...

この理解で正しいでしょうか？
```

#### 非同期処理の確認
```
非同期処理について確認させてください：

1. **状態管理テーブル**: どのテーブルでステータスを管理しますか？
2. **ステータス遷移**: pending → processing → completed のような遷移でよろしいですか？
3. **失敗時の扱い**:
   - リトライしますか？
   - リトライ回数の上限は？
   - 永久失敗時のステータスは？
4. **トリガー**: EventBridge? Lambda? SQS?
5. **実行環境**: Lambda? Fargate?
```

### 3. ロール・権限の確認

#### ロール定義の確認
```
ロール構造について確認させてください：

| ロール名 | 説明 | 主な機能 |
|---------|------|---------|
| [role1] | [説明] | [機能リスト] |
| [role2] | [説明] | [機能リスト] |

各ロールのデータアクセス範囲：
- [role1]: 全[データ]閲覧可能？
- [role2]: 自分の所属する[組織]のみ？
```

#### 権限境界の確認
```
以下のケースでアクセスを許可しますか？

1. [role1]が[role2]のデータを閲覧
2. [role1]が[role2]のデータを編集
3. [role2]が他の[role2]のデータを閲覧
4. [role2]が自分のデータを削除

Row Level Securityで制御しますか？
それともアプリケーションレイヤで制御しますか？
```

### 4. 技術選定の確認

#### ORM/ライブラリの確認
```
[技術名]の採用について確認させてください：

1. **選定理由**: なぜこの技術を選びましたか？
2. **サポート状況**: 開発が活発ですか？長期サポートされますか？
3. **制約**: この技術で実現できない要件はありませんか？
4. **代替案**: 他の選択肢は検討しましたか？
```

#### インフラ構成の確認
```
インフラ構成について確認させてください：

1. **VPC構成**: Lambda/FargateはVPC内？外？
2. **接続方法**: DB接続はData API? コネクションプール?
3. **コスト**: NAT Gateway必要？RDS Proxy必要？
4. **スケーリング**: 想定トラフィック量は？将来の成長予測は？
```

### 5. エッジケースの確認

#### エラーハンドリング
```
エラーハンドリングについて確認させてください：

1. **[エラータイプ]発生時**:
   - リトライしますか？
   - 何回までリトライしますか？
   - リトライ間隔は？
   - 最終的に失敗した場合の処理は？

2. **タイムアウト**:
   - タイムアウト時間は？
   - タイムアウト時の処理は？

3. **部分失敗**:
   - バッチ処理で一部だけ失敗した場合は？
   - 全体をロールバック？失敗分のみリトライ？
```

#### データ整合性の確認
```
データ整合性について確認させてください：

1. **削除時の挙動**:
   - [親テーブル]を削除したら[子テーブル]は？
   - CASCADE DELETE? SET NULL? RESTRICT?

2. **更新時の挙動**:
   - [テーブルA]を更新したら[テーブルB]も更新が必要？
   - トランザクション境界はどこまで？

3. **並行更新**:
   - 楽観的ロック？悲観的ロック？
   - version カラムが必要？
```

## 質問の優先順位

### 高優先度（必ず確認）
1. データ構造の階層
2. 処理フローの全体像
3. ロール定義と権限範囲
4. 外部キー制約とCASCADE動作
5. ステータス管理の取りうる値

### 中優先度（不明瞭な場合確認）
1. NULL許可/NOT NULL制約
2. リトライロジック
3. エラーハンドリング
4. タイムスタンプの種類

### 低優先度（実装時に確認でも可）
1. インデックス設計
2. バリデーションルールの詳細
3. UI/UXの細かい挙動

## 実践例

### 例1: 新機能追加時
```
ユーザー: 「請求機能を追加したい」

質問:
1. 請求の単位は何ですか？（partner単位？client単位？project単位？）
2. 請求期間は？（月次？日次？随時？）
3. 請求金額の計算方法は？（従量課金？固定額？）
4. 請求データの保持期間は？（何年？無期限？）
5. 以下の理解で正しいですか？
   ```
   BILLING_RECORDS:
   - billing_id (PK)
   - partner_id (FK)
   - period (YYYYMM)
   - total_amount (DECIMAL)
   - created_at, updated_at
   ```
```

### 例2: テーブル構造変更時
```
ユーザー: 「sales_textにステータスを追加」

質問:
1. ステータスの取りうる値は？（pending, sent, failed等）
2. デフォルト値は？
3. ステータス遷移のルールは？（pending → sent のみ？戻れる？）
4. 既存データの扱いは？（デフォルト値で初期化？）
5. このステータスで何かトリガーされる処理はありますか？
```

### 例3: 技術選定時
```
ユーザー: 「Kyselyを使いたい」

質問:
1. Kysely Data APIライブラリの開発状況は確認しましたか？
2. サポートが切れるリスクは許容できますか？
3. 代替案として生SQL + Data API SDKは検討しましたか？
4. TypeScript/Python両方で同じ設計パターンを維持できますか？
5. 将来ライブラリを乗り換える可能性はありますか？
```

## チェックリスト

質問を投げる前に確認：
- [ ] 既存ドキュメントを確認したか？
- [ ] 推測で答えられる内容ではないか？
- [ ] 質問が具体的で回答しやすいか？
- [ ] 選択肢を提示しているか？（Yes/Noだけでなく）
- [ ] 図や例を使って理解を示しているか？

## 注意事項

1. **推測は絶対にしない**: 「一般的には〜」で答えない
2. **選択肢を提示**: 「AとBどちらですか？」と具体的に
3. **理解を図示**: 文章だけでなく図や表で確認
4. **段階的に詳細化**: 全体 → 詳細 → エッジケース
5. **一度に質問しすぎない**: 1回の質問は3-5項目まで
