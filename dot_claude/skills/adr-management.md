# Architecture Decision Records (ADR) 運用スキル

## 概要
技術的な意思決定を記録し、将来の開発者が判断の経緯を理解できるようにするADR運用方法

## なぜ重要か

### ADRの価値
- 将来の技術選定の参考になる
- 技術的負債の判断材料になる
- 新規参画者のオンボーディングを加速
- 「なぜこの技術を選んだのか」という疑問を解消
- 代替案の検討履歴が残る

### ドキュメントとの違い

**技術仕様書**: 事実（What/How）を記録
- 「MySQL 3306接続を使用」
- 「NAT Gateway 2個が必要」
- 「Drizzle ORMを使用」

**ADR**: 意思決定（Why）を記録
- 「なぜMySQL 3306接続を選んだか」
- 「なぜNAT Gatewayのコスト増を許容したか」
- 「なぜDrizzle ORMを選んだか」

## ディレクトリ構造

```
project/
├── docs/
│   ├── 03_architecture.md  ← 事実のみ（What/How）
│   └── decisions/
│       ├── README.md  ← ADR一覧
│       ├── ADR-001-mysql-3306-over-data-api.md
│       ├── ADR-002-orm-adoption.md
│       └── ADR-003-vpc-nat-gateway.md
```

## ADR形式

### テンプレート

```markdown
# ADR-XXX: [タイトル]

**状態**: 提案中/採用/却下/非推奨
**日付**: YYYY-MM-DD

## 背景

意思決定が必要になった状況を記載。
複数の選択肢があった場合、それを列挙。

## 決定

何を決定したか、簡潔に記載。

## 理由

なぜこの決定をしたか（箇条書き）:
1. 理由1
2. 理由2
3. 理由3

## トレードオフ

### メリット
- メリット1
- メリット2

### デメリット
- デメリット1
- デメリット2

## 代替案

### 代替案A（却下）

**理由**: なぜ却下したか

**メリット**: この案のメリット

**判断**: 最終的な判断理由

## 影響

システムへの影響を記載:
- 影響1
- 影響2

## 参照

- [関連ドキュメント](../path/to/doc.md)
```

## 2つの記載方式

### 方式1: 独立したADRファイル（推奨）

**用途**: 重要な技術選定、アーキテクチャ決定

**メリット**:
- 意思決定の体系的な管理
- 代替案の検討履歴が残る
- トレードオフが明確
- 技術仕様書がシンプルに保たれる

**使用例**:
```markdown
<!-- docs/03_architecture.md -->
### 2.2 MySQL 3306接続方式

VPC内配置、NAT Gateway 2個使用。

**意思決定の背景**: [ADR-001](./decisions/ADR-001-mysql-3306-over-data-api.md)、[ADR-003](./decisions/ADR-003-vpc-nat-gateway.md)
```

### 方式2: インライン背景記載（軽量な選定）

**用途**: 小規模な技術選定、テーブル設計の背景

**メリット**:
- 簡潔で読みやすい
- ADRファイル増加を抑制
- 設計書内で完結

**使用例**:
```markdown
<!-- docs/04_database_design.md -->
### 4.6 `companies` テーブル

**目的**: 営業先企業の情報を一元管理し、スクレイピング結果を再利用可能にする

**設計意図**:
- 企業情報を正規化し、重複登録を防止
- スクレイピング結果を再利用してコスト削減
- 将来的にclient単位で複数projectをまたがる営業先への送信履歴を表示可能
```

### 使い分けガイドライン

| 観点 | 独立ADR | インライン背景 |
|------|--------|-------------|
| **重要度** | システム全体に影響 | 局所的な影響 |
| **複雑度** | 複数の代替案を検討 | 単一の選択肢 |
| **変更頻度** | 長期的に参照される | 一時的な判断 |
| **例** | VPC構成、ORM選定 | テーブル設計、カラム追加 |

## ADR作成の手順

### Step 1: 意思決定の特定

技術仕様書から意思決定の背景セクションを抽出:
- 「選定理由」「背景」「採用の理由」などのセクション
- 「メリット」「デメリット」「トレードオフ」の説明
- 「代替案」の記載

### Step 2: ADRファイル作成

```bash
# decisions/ディレクトリ作成
mkdir -p docs/decisions

# ADRファイル作成
# 命名規則: ADR-NNN-short-description.md
touch docs/decisions/ADR-001-mysql-3306-over-data-api.md
```

### Step 3: ADR記載

テンプレートに従って記載:
1. 背景: 何が問題だったか、選択肢は何か
2. 決定: 何を決めたか
3. 理由: なぜその決定をしたか（3-5項目）
4. トレードオフ: メリット/デメリットを明記
5. 代替案: 検討したが採用しなかった案とその理由

### Step 4: README.md更新

decisions/README.mdにADR一覧を追記:

```markdown
| ADR | タイトル | 状態 | 日付 |
|-----|---------|------|------|
| [ADR-001](./ADR-001-mysql-3306-over-data-api.md) | MySQL 3306接続を採用 | 採用 | 2025-11-17 |
```

### Step 5: 技術仕様書の更新

意思決定の背景セクションを削除し、ADRへの参照に置き換え:

```markdown
### 2.2 MySQL 3306接続方式

**接続方法**: MySQL 3306ポート経由（VPC内通信）
**認証**: Secrets Manager管理
**ORM**: Drizzle ORM (TypeScript)、SQLAlchemy (Python)

**意思決定の背景**: [ADR-001](./decisions/ADR-001-mysql-3306-over-data-api.md)、[ADR-002](./decisions/ADR-002-orm-adoption.md)
```

## ADRの命名規則

### パターン1: 連番 + 説明（推奨）
```
ADR-001-mysql-3306-over-data-api.md
ADR-002-orm-adoption.md
ADR-003-vpc-nat-gateway.md
```

### パターン2: 日付 + 説明
```
ADR-2025-11-17-mysql-3306-over-data-api.md
ADR-2025-11-17-orm-adoption.md
```

### パターン3: カテゴリ + 連番
```
ADR-db-001-mysql-3306-connection.md
ADR-infra-001-vpc-nat-gateway.md
```

**推奨**: パターン1（連番 + 説明）- シンプルで並び順が明確

## ADRの状態管理

| 状態 | 説明 | 使用例 |
|------|------|--------|
| **提案中** | まだ決定していない | レビュー中の技術選定 |
| **採用** | 決定され、実装済み | 本番稼働中の構成 |
| **却下** | 検討したが採用しなかった | 代替案として記録 |
| **非推奨** | 過去に採用したが今は非推奨 | 技術的負債として記録 |
| **置き換え** | 別のADRに置き換えられた | ADR-XXXで置き換え |

## ベストプラクティス

### DO（推奨）
✅ 簡潔に、箇条書きで記載
✅ トレードオフを明確に記載
✅ 代替案を必ず含める
✅ コスト影響を明記（金額に影響する場合）
✅ ADRは一度記載したら削除しない（状態を「非推奨」に変更）
✅ 技術仕様書とADRを相互参照
✅ 重要度に応じてADRとインライン背景を使い分け

### DON'T（非推奨）
❌ 長文で説明（簡潔に）
❌ 抽象的な記述（「良いから」等）
❌ 主観的な意見のみ（客観的な根拠を含める）
❌ ADRを削除（状態変更で対応）
❌ 技術仕様書にWhy（理由）を重複記載
❌ 小規模な決定をすべてADRにする（インライン背景で十分な場合も）

## インライン背景記載のテンプレート

### テンプレート1: テーブル設計の背景

```markdown
### 4.X `table_name` テーブル

**目的**: このテーブルが解決する課題

**設計意図**:
- 設計方針1
- 設計方針2
- 設計方針3

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| ... | ... | ... | ... |
```

### テンプレート2: 機能実装の背景

```markdown
#### 機能名の実装背景

**選定理由**:
- 技術的メリット1
- 運用上のメリット2
- 保守性・拡張性の観点3

**トレードオフ**:
- メリット: X
- デメリット: Y
- 判断: Z
```

## 既存ドキュメントからADRへの移行

### Step 1: 意思決定箇所の抽出

技術仕様書から以下を抽出:
- 「選定理由」「背景」セクション
- 「メリット」「デメリット」「トレードオフ」
- 「代替案」の記載

### Step 2: 重要度による振り分け

**独立ADRにすべきもの**:
- システム全体に影響する決定
- 複数の代替案を検討した決定
- 長期的に参照される決定

**インライン背景で十分なもの**:
- 局所的な影響の決定
- 単一の選択肢
- 一時的な判断

### Step 3: ADRファイル作成（重要な決定のみ）

抽出した内容をADR形式に変換:
```bash
# サブエージェントに委譲することを推奨
# 複数のADRを一度に作成する場合
```

### Step 4: 技術仕様書の整理

**重要な決定**: ADRへの参照に置き換え
```markdown
**意思決定の背景**: [ADR-001](./decisions/ADR-001-mysql-3306-over-data-api.md)
```

**小規模な決定**: インライン背景として残す
```markdown
**設計意図**:
- 企業情報を正規化し、重複登録を防止
- スクレイピング結果を再利用してコスト削減
```

## トラブルシューティング

### 問題: ADRが多すぎて管理が困難

**解決策**:
- decisions/README.mdにカテゴリ別の目次を作成
- ADRをサブディレクトリに分類（db/, infra/, frontend/等）
- 小規模な決定はインライン背景に変更

### 問題: 過去のADRが古くなった

**解決策**:
- 状態を「非推奨」に変更
- 新しいADRを作成し、古いADRから参照
- 「## 置き換え」セクションに新ADRへのリンクを追加

### 問題: ADRと技術仕様書の整合性が取れない

**解決策**:
- ADRを真実の情報源として扱う
- 技術仕様書には事実のみ記載
- 定期的にADRと技術仕様書の整合性チェック

### 問題: どちらの方式を使うべきか迷う

**判断基準**:
1. システム全体に影響するか？ → Yes: ADR、No: インライン
2. 複数の代替案を検討したか？ → Yes: ADR、No: インライン
3. 長期的に参照される決定か？ → Yes: ADR、No: インライン

## まとめ

ADR運用により:
1. **意思決定の透明性**が向上
2. **技術的負債の判断**が容易に
3. **新規参画者のオンボーディング**が加速
4. **ドキュメントの保守性**が向上

重要な決定は独立したADRとして体系的に管理し、小規模な決定はインライン背景として簡潔に記載することで、ドキュメントの可読性と保守性を両立できます。
